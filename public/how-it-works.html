1. The Data Pipeline

Sunny Pint is a fully static site — no backend at runtime. All the heavy computation happens once in a data pipeline, and the results are served as static files from Cloudflare Pages. The browser does shadow computation client-side at 60fps.

The pipeline runs in six steps, taking about 30 minutes for Norwich or several hours for the whole UK. Every step is idempotent — you can re-run safely.

2. Shadow Computation

Shadows are computed geometrically, not via ray-tracing or raster methods. For each building wall segment, we project a shadow quadrilateral based on the building height and sun position. This produces mathematically exact shadow edges at 60fps.

shadow_length = min(building_height / tan(sun_altitude), 200m)
dlat = -shadow_length × cos(azimuth) / 111,320
dlng = -shadow_length × sin(azimuth) / (111,320 × cos(latitude))

Each wall segment of a building produces a shadow quadrilateral — the four corners are the two endpoints of the wall and their projected positions on the ground. The building's roof footprint, shifted by the shadow offset, adds the "roof shadow". Together these form the complete shadow polygon.

Opacity compositing

When multiple buildings cast overlapping shadows, drawing them all semi-transparently causes opacity stacking — overlaps appear darker. We avoid this by drawing all shadow quads to an opaque offscreen canvas, then compositing the result onto the main canvas with a single alpha blit. This gives uniform shadow darkness regardless of overlap.

Example: The Whalebone, Norwich (52.644, 1.297)

At 4pm in June, the sun is at azimuth ~240° (WSW), altitude ~35°. A 10m building casts a shadow of 10/tan(35°) = 14.3m. At 7pm (altitude ~10°), the same building casts 10/tan(10°) = 56.7m — reaching across the entire beer garden.

3. Measuring Building Heights

Building heights come from Environment Agency LiDAR data: 1-metre resolution laser scans of the ground surface. We use two datasets:

  • DSM (Digital Surface Model) — the top of everything: roofs, trees, cars
  • DTM (Digital Terrain Model) — bare ground with buildings removed

Height = DSM minus DTM. Simple subtraction, no ground estimation needed.

We sample the 90th percentile of pixel values within each building footprint, not the maximum. This captures ridge height on pitched roofs while ignoring chimney and antenna outliers.

4. Terrain Awareness

Elevation adjustment

A building on a hill is effectively taller from the pub's perspective. We store each building's ground elevation (from the DTM) and the pub's elevation, then adjust the effective height:

effective_height = building_height + (building_elevation - pub_elevation)

Terrain horizon occlusion

In hilly areas, a hill between the pub and the sun can block sunlight entirely — even with no building on it. For each pub, the pipeline casts 36 rays (every 10° of azimuth) and records the maximum terrain elevation angle in each direction.

At runtime, the browser compares the sun's altitude at its current azimuth against the horizon profile. If the sun is below the terrain horizon, no shadows are drawn — the pub is fully in terrain shade.

The profile is stored as a compact base64 string (48 characters per pub). Flat-area pubs don't store a profile at all, saving space.

5. The Porthole View

The porthole is a Canvas 2D rendering, not a map library. It draws real map tiles from CARTO as a background, then overlays building polygons and shadow geometry on top.

7. Optimisations

Height-dependent building filter

Not every building within 300m of a pub needs to be in the tiles. A short garden shed 250m away can never cast a shadow that reaches the pub. The maximum shadow reach depends on the building height:

reach = porthole_radius + min(height / tan(3°), 200m)

6m (shed)114m188m
8m (default)153m227m
10m (two-storey)191m265m
15m+ (tall)200m (capped)274m

This filter reduces the number of buildings in tiles by ~30% for Norwich, with proportional savings in file size and download time.

Pre-exploded vector tiles

We tried PMTiles (a single-file spatial archive using HTTP range requests) but Cloudflare Pages strips the Content-Length header when compressing responses, which breaks PMTiles. Instead, we pre-explode the archive into individual z14 .pbf files. Each pub needs 1-4 tile fetches (~10-100 KB each), and the browser caches them via the service worker.

Polygon stripping

The full building polygon for each pub (20-50 coordinate pairs) was 70% of pubs.json. The frontend only needs the centroid for matching, so we pre-compute it in the pipeline and ship just two numbers (clat, clng). This reduced the file from 11.3 MB to 6.3 MB (44% saving).
